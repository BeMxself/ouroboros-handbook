# 潮汐栈

> 快速构建、融合开发、快速迭代

潮汐栈是一款面向专业团队的低代码平台，重点不是“替代开发”，而是帮助团队把低代码能力、专业开发能力和长期运维能力组合起来，持续交付和演进企业应用。

它想解决的，也不只是“把第一版做快”。在很多企业项目里，真正持续消耗团队成本的，往往是初版之后不断变化的需求、交付前后的反复调整，以及上线后的维护和对接工作。潮汐栈希望把这些阶段放在同一条交付链上一起考虑。

## 平台定位

潮汐栈希望解决的是企业项目里最常见的三类割裂：

- 业务开发和专业开发割裂
- 前期原型和后续交付割裂
- 开发交付和后续运维割裂

因此它同时强调三件事：

- 用模型和配置提升标准业务功能的交付效率
- 用扩展机制承接复杂场景和个性化需求
- 用统一的交付与维护方式支撑长期演进

## 平台设计哲学

潮汐栈的设计哲学可以概括为四点：

- 不只追求初版开发速度，而是追求需求持续变化下的整体交付效率
- 不把团队锁在平台舒适区里，而是允许低代码与 Pro Code 接力开发
- 不把平台做成封闭孤岛，而是强调对接能力、集成能力和可持续运维
- 不要求团队先学习大量专有概念，而是尽量贴近通用技术、行业规范和常见工程做法

这也是为什么手册会同时展开低代码、融合开发和运维管理三条主线。潮汐栈并不把它们看作彼此割裂的三套体系，而是看作同一套平台在项目全生命周期中的不同侧面。

## 核心能力

- 基于模型的业务开发方式
- 页面、逻辑流、审批流等可视化能力
- 动态表单、文档管理、文件上传等业务支撑能力
- 面向跨系统协同的接口、消息与实时集成能力
- 调度任务、消息队列、WebSocket 等集成与实时能力
- 组件库、Widget、版本治理、数据源与环境管理等平台能力
- 面向复杂场景的前后端扩展机制
- 支持版本管理与持续交付
- 支持应用部署、升级和长期维护

如果你想先看一眼平台当前能力覆盖到什么程度，再决定走哪条学习路径，可以直接查看 [平台能力地图](./reference/platform-capability-map)。
