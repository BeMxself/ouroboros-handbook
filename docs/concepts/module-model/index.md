# 模块模型

适用对象：负责大型应用信息架构、模块拆分和功能编排的人。

模块模型用来定义系统的功能分组和层级关系。它不是具体业务能力本身，而是承接“这些能力应该怎么被组织起来”的装配模型。

## 它解决什么问题

- 让大量业务对象和能力有稳定的归属结构
- 让菜单、权限和业务功能围绕同一个模块树管理
- 让应用从“若干孤立模型”变成“有边界、有层次的系统”

## 核心组成

- 模块树：模块节点及其层级关系
- 归属关系：业务模型、菜单、权限等挂载到哪个模块
- 编排信息：排序、命名、说明和展示结构

## 与其他模型的关系

- [业务模型](../business-model/) 往往按模块归类
- [菜单模型](../menu-model/) 和 [权限模型](../authority-model/) 常与模块树协同管理
- 模块模型不直接承载页面或接口细节，但影响整个应用的组织方式

## 典型使用场景

- 把“主数据、采购、销售、库存、财务”等能力拆成清晰模块
- 管理大型项目里的菜单和权限归属
- 让应用交付和后续迭代有更稳定的信息架构
