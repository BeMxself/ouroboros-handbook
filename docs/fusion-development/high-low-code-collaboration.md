# 高低开协同模式

高低开协同的关键，不是把所有东西都改写成代码，而是让“低代码负责稳定结构，代码负责定制能力”，并且让两者能够接力，而不是互相推翻。

这也是潮汐栈和很多“先低代码、后重写”的做法最大的差异之一：平台希望复杂需求被扩展点承接之后，业务团队仍然能继续留在平台主线上工作。

## 这页主要解决什么

这页主要回答三类问题：

- 哪些工作适合继续留在低代码里
- 哪些工作应该交给专业开发承接
- 两类角色在同一个应用里应该怎么交接，才不至于互相踩边界

如果你还在判断“这件事到底该不该写代码”，建议先看 [何时用低代码，何时写代码](./when-to-use-code)。

## 推荐分工

| 角色 | 更适合负责什么 | 不建议优先负责什么 |
| ---- | -------------- | ------------------ |
| 低代码开发人员 | 业务模型、标准页面、流程串联、权限配置、通知和组织协同 | 复杂前端交互、深度集成、平台底层扩展 |
| 专业开发人员 | 复杂页面、特殊交互、平台扩展、外部系统深度集成、特殊访问控制 | 只靠代码重做整条标准业务链 |

这不是绝对隔离，而是一个更稳的默认分工。

## 协同通常发生在哪些交接点

### 业务模型已经稳定，但页面体验超出标准能力

这时通常由低代码先把模型、菜单和标准流程搭好，再由专业开发通过 [前端融合开发方式](./frontend-extension-paths) 接手复杂页面、登录页或布局替换。

### 标准流程能跑通，但某个能力需要程序化扩展

例如：

- 新字段类型
- 新表达式能力
- 新脚本上下文
- 新逻辑流节点

这时更适合由专业开发通过 [平台扩展点总表](./extension-points) 中的后端扩展点补能力，再让业务开发人员继续在平台内配置和使用。

### 平台主线已经覆盖 80% 以上，剩下是少量定制点

这是最理想的协同状态。

这时不应该因为少数复杂点，就把整套交付方式全部切成纯代码；而应该只把超出平台舒适区的部分抽出来做扩展。

## 一条推荐的协同顺序

大多数项目里，建议按下面顺序协同：

1. 先由低代码把业务对象、标准页面和核心流程骨架立起来
2. 再识别真正卡住交付的边界点
3. 由专业开发通过明确扩展点承接这些边界点
4. 扩展完成后，把能力尽量回收到平台主线中继续复用
5. 业务开发人员继续在平台里接手配置、交付和迭代

这条顺序的核心价值在于：把“代码扩展”变成对平台主线的补充，而不是替代。

## 三条很重要的协同原则

### 先低代码，再扩展

优先先把业务骨架和标准能力跑通，再决定哪些地方真的需要代码。

### 扩展要通过平台入口接入

无论是前端还是后端扩展，尽量都通过平台已有扩展点接入，而不是绕开平台单独实现一套运行链路。

### 扩展完成后，最好还能回到平台主线

理想状态不是“业务团队从此只能找开发改代码”，而是“开发把复杂能力补进平台后，业务团队还能继续配置和使用”。

## 常见协同模式

### 模式一：低代码主导，代码补点

适合大多数标准业务项目。

表现通常是：

- 模型、页面、流程主要留在平台
- 只对个别页面、个别字段或个别接口做扩展

### 模式二：代码主导，低代码承接标准部分

适合复杂度较高、但仍希望保留平台统一交付方式的项目。

表现通常是：

- 专业开发先补好关键扩展点
- 业务开发再用这些能力去做剩余交付

### 模式三：平台内外协同

适合需要外部系统深度集成的场景。

表现通常是：

- 平台负责业务骨架、表单、流程和入口
- 外部系统或自定义服务负责特殊计算、专业界面或外部对接

## 常见误区

### 误区一：遇到复杂需求就整体退出平台

这通常会让前期低代码成果无法复用，也会让后续运维和交付链重新分裂。

### 误区二：什么都想先留在低代码里硬扛

如果某个需求已经明显超出标准页面、标准流程和标准配置边界，继续硬扛往往只会让系统越来越难维护。

### 误区三：扩展点做完后，不再回到平台主线

这样会让业务团队逐渐失去接手能力，最后变成“平台只剩壳，实际交付全靠代码”。

## 做到什么算协同方式选对了

- 业务骨架仍然留在平台主线上
- 复杂点被明确定位到具体扩展点
- 扩展后能力仍能被业务开发人员继续使用
- 没有因为少量复杂需求把整条交付链推翻
