# 集成、自动化与实时能力

很多团队在进入融合开发时，第一反应是“我要自己写一套集成框架”。但在潮汐栈里，调度、消息和实时通信已经有现成底座，很多需求可以先站在平台能力上做。

这页主要帮助你判断三类常见能力怎么选：

- 定时或一次性任务怎么做
- 异步解耦和消息驱动怎么做
- 实时状态推送和在线通知怎么做

## 三类能力分别解决什么问题

### 调度能力

`ouroboros-scheduler` 解决的是“任务什么时候触发”。

当前平台已经提供：

- Cron 调度和一次性任务调度
- 任务来源发现与聚合
- Quartz 默认调度实现
- 调度任务与 LogicFlow 的执行桥接

如果你的核心问题是“定时执行一个业务动作”或“在未来某个时间触发流程”，先看调度能力。

### 消息队列能力

`ouroboros-message-queue` 解决的是“消息如何异步发送和消费”。

当前平台已经提供：

- 统一的 `MQBridge`
- 生产者、消费者抽象
- 绑定配置与多来源聚合
- 默认 RabbitMQ 适配
- LogicFlow 消费和发送接入

如果你的核心问题是“解耦调用链”“异步处理”“把事件发给别的处理者”，先看消息队列能力。

### WebSocket 实时能力

`ouroboros-websocket` 解决的是“消息怎么实时推送到在线用户或会话”。

当前平台已经提供：

- 基于 STOMP 的发送与订阅接口
- 广播、用户定向和会话级消息语义
- 连接鉴权
- 通道拦截和会话管理
- Spring Boot 自动装配

如果你的核心问题是“前端需要及时收到状态变化”，先看 WebSocket，而不是先做轮询。

## 一张简单判断表

| 需求核心                           | 优先能力                       |
| ---------------------------------- | ------------------------------ |
| 到点触发任务、定时跑流程、延迟执行 | 调度                           |
| 异步处理、解耦上下游、事件分发     | 消息队列                       |
| 在线状态推送、进度更新、实时通知   | WebSocket                      |
| 既要定时触发，又要异步分发         | 调度 + 消息队列                |
| 后端状态变化后要立即反馈前端       | 消息队列或业务处理 + WebSocket |

## 推荐的组合方式

### 组合一：调度 + LogicFlow

适合定时报表、定时同步、定时巡检、周期提醒这类任务。

在这种场景里，调度负责触发时机，LogicFlow 负责实际业务执行，职责边界比较清晰。

### 组合二：消息队列 + LogicFlow

适合订单事件、审批后续动作、外部系统回调后续处理、异步通知等链路。

在这种场景里，消息队列负责解耦和投递，LogicFlow 负责把消息转成平台内的业务处理。

### 组合三：业务处理 + WebSocket

适合任务状态变更、运行进度更新、操作反馈、在线提醒等体验类需求。

在这种场景里，WebSocket 不负责业务落地，而是负责把结果及时推给当前在线用户。

### 组合四：调度 + 消息队列 + WebSocket

适合“后台任务异步执行，前端实时看到状态”的场景。

这也是平台内部某些运行控制能力已经采用的组合思路，所以通常没必要从零再拼一套同类基础设施。

## 什么时候优先复用平台现成能力

- 任务本质上还是“触发一个平台内业务流程”
- 消息语义可以落在平台已有绑定和生产消费模型里
- 实时能力只需要广播、用户级或会话级推送
- 团队更关心尽快交付业务，而不是自建基础设施

这时优先复用平台底座，维护成本会明显更低。

## 什么时候应该做代码扩展

- 需要接入新的消息中间件或定制更深的中间件策略
- 需要超出当前调度抽象的特殊任务编排能力
- 需要更复杂的实时协议、网关或连接治理方式
- 需要把这些能力和外部系统做深度程序化对接

进入扩展时，也建议沿着平台已有边界做，而不是绕开抽象层直接写死具体实现。

## 写代码时优先落在哪些边界

### 调度

优先沿着 `Scheduler`、`ScheduleTaskProvider`、`ScheduleTaskIntegrator` 这些边界扩展。

### 消息队列

优先沿着 `MQBridge`、绑定模型、producer/consumer 抽象和中间件适配点扩展。

### WebSocket

优先沿着发送接口、订阅接口和拦截器链扩展，而不是把业务协议塞进运行时底座。

## 一个实用原则

如果你现在要解决的是“业务如何被触发、解耦和反馈”，先复用平台现成能力。

如果你现在要解决的是“平台底座本身不够用，必须扩展通信或调度机制”，再进入代码扩展。
