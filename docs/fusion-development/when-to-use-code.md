# 何时用低代码，何时写代码

这页的目标不是鼓励尽快写代码，而是帮你减少两类误判：

- 明明平台已经能做，却过早进入定制开发
- 明明已经被平台边界卡住，却还在低代码里硬拧

## 优先留在低代码主线的场景

- 需求主要是业务对象、字段、表单、列表和基础流程
- 页面结构基本符合平台已有能力
- 权限、通知、配置项等需求主要是标准配置

## 应该优先先试低代码的典型任务

| 任务 | 通常先用什么 |
| ---- | ---- |
| 建业务对象、字段、列表、详情页 | 数据模型 + 业务模型 |
| 做标准表单、查询页、审批流 | 低代码页面 + 逻辑流 |
| 做基础权限、通知、配置项 | 平台现成能力 |
| 做一般性的业务规则拼装 | 表达式 + 脚本 + 节点编排 |

## 应该考虑写代码扩展的场景

- 页面交互有明显定制需求
- 需要替换登录页、布局或接入独立前端应用
- 需要自定义字段、表达式、脚本能力或逻辑流节点
- 需要和外部系统做深入程序化集成

## 什么时候基本已经进入融合开发

出现下面任一特征时，通常就不该继续把问题当成“纯低代码配置”来解决：

- 你已经需要新的平台扩展点，而不是现有能力的普通配置
- 你要把一类能力沉淀给多个应用或多个团队复用
- 你需要接入外部系统 SDK、企业中台、消息总线或定制认证
- 业务关键能力已经需要靠自定义代码来保证可维护性和可测试性

## 一个实用的判断矩阵

| 判断问题 | 更偏低代码 | 更偏写代码 |
| ---- | ---- | ---- |
| 问题核心是“怎么描述业务”还是“怎么实现程序行为” | 描述业务 | 实现程序行为 |
| 结果是“一次配置”还是“沉淀一种新能力” | 一次配置 | 新能力 |
| 需求主要在页面/模型/流程组合内可解决吗 | 可以 | 不可以 |
| 团队后续主要是业务人员维护，还是专业开发者维护 | 业务人员 | 专业开发者 |

## 一个实用判断

如果问题的核心是“如何描述业务”，优先用低代码。

如果问题的核心变成“如何实现特定程序行为”，就应该进入融合开发。

## 一条更稳的推进方式

1. 先用低代码把最小可用版本跑通。
2. 明确真正卡住你的边界点是什么。
3. 只把被卡住的那一段抽到融合开发里处理。
4. 让扩展结果重新回到平台能力中被业务继续消费。

这样做比“一上来全写代码”更容易维持潮汐栈的协同优势。

## 降低误判的小建议

- 先尝试用平台现有模型、页面和流程把最小可用版本跑通
- 只有当关键需求被平台边界卡住时，再决定写代码
- 一旦进入代码扩展，尽量让扩展点保持聚焦，避免把整条业务链都拉出平台体系
