# 融合开发总览

融合开发关注的不是“要不要写代码”这一个问题，而是如何在不破坏平台统一性的前提下，把低代码能力和专业开发能力组合起来。

它背后的判断很简单：企业项目不可能永远停留在平台现成能力的舒适区里。如果一个平台一旦遇到复杂需求就只能整块推倒重来，或者把已有低代码成果和代码扩展彻底割裂开，那平台越用越重，团队也越容易失去继续投入的意愿。

## 它解决什么问题

- 低代码无法完整表达的复杂界面与交互
- 低代码无法直接覆盖的后端能力扩展
- 平台模型与定制代码之间的协同边界
- 复杂需求、外部系统对接和平台统一性之间的平衡

## 融合开发的核心原则

潮汐栈强调的不是“低代码或代码二选一”，而是：

- 能留在低代码里的，尽量留在低代码里，保持交付效率和统一性
- 超出平台舒适区的部分，用扩展点承接，而不是推倒已有成果
- 扩展完成后，业务开发人员最好仍然能继续在平台里接手配置、交付和维护

## 典型扩展方向

- 前端扩展：微前端、自定义组件、自定义登录页、自定义布局
- 后端扩展：字段、表达式、脚本、逻辑流节点、定制能力接入

## 开始写代码前，先确认哪些能力已经内建

很多需求看起来像“要写代码”，但平台其实已经内建了不少可以直接复用的能力，例如：

- 调度任务、消息队列、WebSocket 等集成底座
- 文件上传、文档管理、动态表单等业务支撑能力
- Widget、平台组件库、用户仪表板等前端承载能力

建议先看 [平台能力地图](../reference/platform-capability-map)，再决定是否真的需要进入扩展开发。

如果你关心的是调度、消息和实时链路怎么协同，可以继续看 [集成、自动化与实时能力](./integration-automation-and-realtime)。

如果你关心的是首页、工作台、Widget 和前端复用能力怎么分层，可以继续看 [工作台、Widget 与平台组件库](./workbench-widgets-and-component-libraries)。

如果你关心的是微前端页面、自定义登录页、布局组件和组件库该怎么选，可以继续看 [前端融合开发方式](./frontend-extension-paths)。

如果你关心的是自定义登录页应该怎么接微前端与认证能力，可以继续看 [自定义登录页与认证接入](./custom-login-page-and-auth-integration)。

如果你关心的是后端能力如何打成扩展包，或开发平台模块如何走 Maven 脚手架，可以继续看 [创建后端扩展包](./backend-extension-packages) 和 [开发模块与 Maven 脚手架](./development-module-scaffold-and-maven-skeleton)。

如果你关心的是数据源、环境、容器和版本治理这些开发平台辅助能力，可以继续看 [开发平台辅助能力](./development-platform-support-capabilities)。

如果你关心的是权限模型已经超出普通 RBAC 配置，开始涉及主体接入、Claims 聚合和字段/数据级访问控制，可以继续看 [权限、主体与访问控制设计](./authority-subject-and-access-control-design)。

## 进入融合开发前先确认

在开始写代码前，建议先把这三件事判断清楚：

- 这个需求是否真的超出了低代码内建能力
- 你要扩展的是前端体验、后端能力，还是两者协同边界
- 扩展完成后，业务开发人员是否还能继续在平台内接手配置和交付

## 做到什么算方向判断正确

- 你能说清楚为什么这个需求不能只靠低代码完成
- 你能定位应该进入哪个扩展点，而不是在多个入口里反复试错
- 你知道扩展后的能力如何继续回到平台主线中被复用
- 你没有因为局部复杂需求，把整套低代码交付方式整体抛弃
