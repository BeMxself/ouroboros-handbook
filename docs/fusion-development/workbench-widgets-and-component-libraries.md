# 工作台、Widget 与平台组件库

这页适合正在做首页、看板、门户入口或前端能力复用的团队。

这类需求很容易一上来就走到“我要做一个独立前端应用”，但潮汐栈当前已经有三层不同能力，可以先判断是否真的需要走到微前端：

- 工作台：面向最终用户的首页和入口组织
- Widget：面向页面拼装的小组件能力
- 平台组件库：面向多应用复用的前端能力分发

如果你当前还在“组件库”和“平台组件库”之间犹豫，可以先抓住一个最核心的区分：

- 组件库：更偏页面内复用
- 平台组件库：更偏跨应用分发与治理

## 先把三层能力分开

### 工作台

工作台解决的是“用户进入平台后先看到什么、怎么进入常用操作”。

当前平台里的 Widget 体系已经包含：

- 用户工作台页面
- 工作台模板管理
- 快捷入口类 Widget
- 用户信息类 Widget

如果你的目标是做“我的工作台”“部门看板”“常用入口首页”，先从工作台思路开始，而不是先做独立子应用。

### Widget

Widget 解决的是“页面里的一块内容怎么被复用和配置”。

当前平台已经内建了一组常见 Widget 定义，包括：

- 表格
- 列表
- 柱状图、折线图、饼图
- 通用图表
- 快捷入口
- 用户信息

同时平台还提供：

- Widget 定义加载
- Widget Schema 管理
- 基于生成器的配置表单
- 用户仪表板对 Widget 的装配和保存

如果你的目标是拼一个看板、门户页或仪表板，通常先看 Widget 就够了。

### 平台组件库

平台组件库解决的是“怎么把一批前端能力分发给多个应用复用”。

当前 `ouroboros-dev-platform-component-lib` 已经支持：

- 上传和检查 ZIP 组件库包
- 安装、卸载和版本治理
- `components.manifest.json` 清单校验
- 公共路径和组件定义冲突检测
- 应用按需导入和移除组件库
- 通过版本控制记录组件库变更

如果你的目标是把一组渲染器、表单项、动作或编辑器插件稳定分发给多个应用，平台组件库才是更合适的层级。

## 一张复用层级判断表

| 当前复用范围 | 优先方式 | 说明 |
| ------------ | -------- | ---- |
| 只在一个门户页里拼装卡片和入口 | Widget | 页面拼装层 |
| 只在一个应用里多页复用某类控件 | 组件库 | 页面内复用层 |
| 要分发给多个应用统一安装和升级 | 平台组件库 | 平台分发治理层 |
| 要做整页独立交互 | 微前端 | 页面承载层 |

## 一张简单判断表

| 当前目标                                 | 优先能力   |
| ---------------------------------------- | ---------- |
| 做一个用户首页、常用入口页、个人工作区   | 工作台     |
| 拼装图表卡片、列表卡片、快捷入口和数据块 | Widget     |
| 让多个应用复用一批前端组件或扩展点       | 平台组件库 |
| 单页交互特别复杂、需要独立前端运行方式   | 微前端     |

## 常见组合方式

### 组合一：工作台 + Widget

这是最自然的一条路径。

工作台负责页面承载，Widget 负责把表格、图表、快捷入口和用户信息拼进来。对很多管理后台和门户首页来说，这已经足够。

### 组合二：平台组件库 + Widget

适合你们希望把一批标准化前端能力交给多个应用共享的场景。

这时平台组件库负责分发复用能力，Widget 负责在具体页面里承接展示。

### 组合二点五：组件库 + 平台组件库

适合你们已经有一批较稳定的页面内组件，准备把它们升级为跨应用共享能力的场景。

这时可以把“组件设计约束”先收敛在组件库层，再把成熟能力打包进平台组件库，而不是一开始就把所有试验性组件都放进平台级分发。

### 组合三：平台组件库 + 微前端

适合你们既需要平台级复用能力，又需要承接少量复杂独立页面的场景。

这时不要把所有需求都塞进微前端里。可以把通用能力沉淀进平台组件库，把真正复杂的页面交给微前端。

## 什么时候先不要急着写前端代码

- 只是要做一个首页、工作台或快捷入口页
- 只是要把列表、图表和卡片组合成看板
- 只是要把同类组件复用到多个应用
- 只是要给现有页面补少量前端扩展点

这些需求里，很多都还没有越过平台边界。

## 什么时候应该升级到平台组件库

- 你们已经确定一批组件会被多个应用复用
- 这批能力需要版本治理，而不是每个应用各拷一份
- 这批前端能力需要通过正式入口被导入、升级和移除

如果只是一个应用里偶尔复用几段前端代码，还不一定要立即做成平台组件库。

## 什么时候先停在组件库就够了

- 这批组件只服务单个应用
- 组件 API 还在快速变化
- 还不想引入版本治理和安装卸载流程

这时先把边界和 API 收敛在组件库层会更稳。

## 什么时候应该直接进入微前端

- 页面本身就是一个独立子系统
- 交互复杂度明显超出 Widget 和 Schema 组合能力
- 需要独立前端工程、独立路由或更强的运行时控制

走到这一步时，再去看微前端会更顺，而不是在工作台或 Widget 层反复硬撑。

## 一个实用判断原则

如果你要解决的是“页面怎么被拼出来”，先看工作台和 Widget。

如果你要解决的是“能力怎么在一个应用里被多页复用”，先看组件库。

如果你要解决的是“能力怎么被多个应用复用”，再看平台组件库。

如果你要解决的是“要不要独立做一个前端子应用”，最后才看微前端。
