# 从需求到交付

第一次做业务功能时，最容易遇到的问题不是“不会点哪个按钮”，而是不知道应该按什么顺序使用平台能力。

## 推荐交付路径

1. 明确业务对象和核心流程
2. 创建应用并建立业务模型
3. 完成页面和表单的基础结构
4. 增加逻辑流、审批流或通知能力
5. 补齐身份、访问边界和组织结构关联
6. 做一次完整业务闭环验证

## 每一步至少要产出什么

- 明确业务对象和核心流程：说清楚要管理什么数据、谁发起、谁处理、结果如何流转
- 创建应用并建立业务模型：平台里已经有应用骨架和核心实体
- 完成页面和表单的基础结构：业务人员已经能录入、查看和检索基础数据
- 增加逻辑流、审批流或通知能力：关键业务动作已经能被系统驱动
- 补齐身份、访问边界和组织结构关联：不同身份看到的入口、数据范围和可执行操作已经分开
- 做一次完整业务闭环验证：从创建数据到处理结果已经能走通一遍

## 一个简单判断原则

如果某个需求主要是“定义数据、配置页面、串业务流程”，优先停留在低代码主线。

如果某个需求已经变成“需要写特定前后端代码才能成立”，再进入 [融合开发](../fusion-development/)。

## 完成后下一步看哪里

- 如果你已经完成了第一个最小业务闭环，继续扩展主题时可回到 [低代码开发总览](./overview)
- 如果你在页面、交互或集成能力上碰到明显边界，再切换到 [融合开发](../fusion-development/)
