# 低代码开发总览

低代码开发面向的是“以交付业务功能为第一目标”的开发过程。它不是简单拖拽，而是基于平台提供的模型、页面、流程和权限能力，尽可能用统一方式完成大部分标准功能。

这条路径要解决的，也不只是“把第一版更快做出来”。在很多真实项目里，需求会在开发、验收和上线前后不断调整，如果每次变化都要迅速退回到大量手写代码，低代码的价值就会明显缩水。潮汐栈希望把更多反复变更的成本留在模型和配置层解决。

## 这条路径解决什么问题

- 用统一模型组织应用结构
- 用配置和可视化能力减少重复实现
- 让业务开发与后续二开扩展保持连续
- 让需求迭代时尽量不必频繁推翻已有交付成果

## 常见工作内容

- 创建应用
- 建立业务模型
- 配置页面、表单与菜单
- 设计逻辑流或审批流
- 配置角色权限、组织架构、通知和配置项
- 结合动态表单、文件上传、文档管理等支撑能力补齐交付细节

## 看到什么算是入门完成

完成这一阶段后，你通常应该已经具备下面这些结果：

- 能创建并打开一个应用
- 能围绕一个业务对象建立基础模型
- 能把列表、表单、流程和权限串成最小闭环
- 知道哪些需求还能继续留在低代码里做，哪些需求需要切到代码扩展

## 除了主线，还能补哪些能力

如果你的业务场景需要轻量数据采集、附件上传、文档库或自定义页面模板，未必一开始就要进入代码扩展。建议先补看 [平台能力地图](../reference/platform-capability-map)，确认平台现成能力是否已经足够。

## 什么时候需要跳到融合开发

如果你遇到以下情况，通常意味着应该开始看 [融合开发](../fusion-development/)：

- 页面交互明显超出平台标准能力
- 需要自定义前端组件、布局或登录页
- 需要自定义字段、表达式、脚本或逻辑流节点
- 需要和外部系统做更深度的程序化集成

理想状态不是“低代码失败了，所以开始写代码”，而是“低代码先完成大部分标准部分，再由代码接力承接平台舒适区之外的需求”。
