# 页面交付：标准页面、动态表单与 UI Schema

这页适合已经进入低代码交付阶段，但正在犹豫“这个页面到底该走业务模型标准页面、动态表单，还是自定义 UI Schema”的团队。

很多项目不是不会做页面，而是容易在这几个选择之间反复摇摆：

- 核心对象到底要不要先建业务模型
- 轻量采集是不是直接上动态表单更快
- 标准页面不太合适时，是不是先用 UI Schema 覆盖
- 什么情况下还不需要直接切到微前端或自定义前端

在潮汐栈里，这三条路线都可以成立，但适用边界并不一样。

## 先理解三种页面承载方式

### 标准业务模型页面

标准页面适合围绕稳定业务对象交付列表、表单、详情等主页面。

如果你的对象是订单、合同、客户、设备、报销单这类核心业务实体，优先还是先建业务模型，再使用平台标准页面能力。

这条路线的优点是：

- 业务对象更稳定
- 和权限、流程、接口、菜单更容易形成统一闭环
- 后续要补逻辑流、审批流和扩展能力也更顺

### 动态表单

动态表单适合轻量采集、登记、台账、问卷、试点表单这类需求。

从当前模块实现看，它不仅支持定义字段，还支持比较完整的生命周期：

- `未部署`
- `已部署`
- `已下线`
- `已归档`

并且在部署后可以生成：

- 表单页
- 表单页面入口
- 表格页
- 表格页面入口

这意味着动态表单不只是一个“临时录入控件集合”，而是一条可独立运行的轻量页面交付路径。

### 自定义 UI Schema

自定义 UI Schema 适合“页面入口已经明确，但标准页面结构不够用，还不想直接进入前端代码开发”的场景。

当前 `ui-schema-management` 主要按以下对象治理：

- `path`
- `schema`
- `enabled`

也就是说，它更像“把某个路径上的页面实现当作可管理资产维护”，适合做：

- 页面模板覆盖
- 特定页面的运行时定制
- 系统设置类或辅助页面治理
- 标准入口下的结构化界面调整

如果你需要的是“按路径替换页面实现”，UI Schema 会比直接写前端更轻。

## 一张简单判断表

| 当前问题                                 | 优先路线             |
| ---------------------------------------- | -------------------- |
| 核心业务对象要长期维护和持续演进         | 标准业务模型页面     |
| 先快速上线一个采集、登记或试点页面       | 动态表单             |
| 页面入口已经有了，但展示结构需要单独定制 | 自定义 UI Schema     |
| 标准页面不够，但仍属于结构化业务界面     | 标准页面 + UI Schema |
| 表单变化快，后续可能再沉淀成正式模型     | 动态表单，后续再迁移 |
| 页面交互明显超出 Schema 和标准组件边界   | 切到融合开发         |

## 为什么这三种方式不要混着起步

它们分别解决的是不同层级的问题：

1. 标准业务模型页面解决“核心业务对象如何稳定交付”
2. 动态表单解决“轻量采集和快速上线如何先跑起来”
3. 自定义 UI Schema 解决“某个页面路径如何被定制化承载”

如果一开始就把同一个对象同时放进这三条路线里维护，后面很容易出现：

- 数据口径重复
- 页面入口混乱
- 权限和菜单边界不清
- 后续迁移成本变高

更稳妥的做法通常是先选主路线，再为少量边缘场景补第二种能力。

## 常见组合方式

### 组合一：业务模型 + 标准页面

这是最标准、也最推荐的业务交付主线。

适合核心对象稳定、后续还要接权限、流程、接口和运维管理的场景。

### 组合二：动态表单 + 文件上传

适合先快速承接活动报名、巡检记录、临时登记、台账采集这类场景。

动态表单负责采集结构，上传能力负责附件承载，通常能很快形成可用页面。

### 组合三：标准页面 + 自定义 UI Schema

适合主能力仍然建立在业务模型上，但局部页面展示方式需要覆盖或增强的场景。

这时可以保留标准交付主线，只在个别路径上用 UI Schema 调整页面。

### 组合四：动态表单 + 审批流

适合“先收集数据，再进入审批”的轻量流程场景。

从平台能力边界看，审批流本身也和动态表单存在集成基础，所以这类场景通常不必一开始就做完整前后端定制。

## 推荐推进顺序

如果你正在为一个新需求决定页面交付方式，通常建议按下面顺序判断：

1. 先判断这个对象是不是核心业务模型
2. 如果是核心对象，优先从业务模型和标准页面开始
3. 如果只是轻量采集或试点，优先评估动态表单
4. 如果标准页面不合适，但仍属于结构化业务界面，再评估 UI Schema 覆盖
5. 如果页面体验已经明显超出 Schema 能力，再进入融合开发

这样推进，既能避免过早写代码，也能避免把临时页面做成长期主路径。

## 一个很实用的边界原则

### 动态表单不是核心业务建模的替代品

它很适合快速上线，但不一定适合长期承载复杂核心业务域。

### UI Schema 不是所有前端定制的终点

它很适合结构化、声明式页面，但不适合无限承载复杂交互。

### 标准页面也不是必须一成不变

当标准页面已经覆盖大部分需求时，先局部补 UI Schema，通常比直接切到微前端更稳。

## 什么时候留在低代码里就够了

- 页面仍然以列表、表单、详情、审批页这类结构化界面为主
- 动态表单已经能承接需求变化较快的采集场景
- 自定义 UI Schema 只用于页面覆盖、模板调整和运行时定制
- 页面交付的核心目标仍然是快速实现业务闭环

这时优先留在低代码主线，通常会更省成本。

## 什么时候应该切到融合开发

- 需要自定义复杂交互、专用可视化或重前端体验
- 需要明显超出标准组件和 Schema 表达能力的页面逻辑
- 需要把动态表单或自定义页面与外部系统做更深程序化联动
- 需要在前端层引入专用框架、微前端或独立交互体系

这时再进入 [融合开发](../fusion-development/) 会更合适。
