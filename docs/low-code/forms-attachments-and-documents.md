# 表单、附件与文档能力

这页适合已经走上低代码主线，但遇到“除了业务模型和流程，还要怎么承接录入、附件和资料沉淀”的团队。

很多项目做到一半才发现，真正消耗时间的并不只是模型和流程，还包括这几类需求：

- 轻量采集和临时登记
- 业务记录上的附件上传
- 团队共享资料和制度文档管理
- 某些页面模板需要在运行时单独维护

潮汐栈目前已经内建了对应能力，不一定一开始就要进入代码扩展。

## 先判断该用哪一类能力

### 业务模型

适合字段结构相对稳定、要长期进入主业务流程的对象。

如果你的目标是订单、合同、报销单、客户、设备这类核心业务对象，优先还是从业务模型开始，而不是先建动态表单或文档库。

### 动态表单

适合轻量采集、登记、台账、问卷、临时活动收集这类需求。

当前平台里的 `dynamic-form` 支持：

- 定义表单及字段
- 部署、下线、重新部署、归档等生命周期
- 生成表单页和表格页
- 对已部署表单做数据增删改查
- 批量导入表单数据

如果你的需求重点是“先把表单收集跑起来”，而不是先建立完整业务域模型，动态表单通常是更轻的入口。

### 文件上传与附件

适合“某条业务记录需要挂附件”的场景，例如合同附件、报销票据、项目资料、图片或导入文件。

当前平台里的 `ouroboros-file-upload` 提供的是统一上传底座，包括：

- 统一 Web 上传入口
- 临时文件、正式文件和分片上传
- 文件访问地址生成
- 默认文件系统存储实现
- 临时文件和分片文件清理

如果你的需求只是给业务记录补附件能力，通常先复用上传能力和附件字段就够了，不需要单独建文档管理方案。

### 文档管理

适合“资料本身就是一个独立资产”的场景，例如制度库、项目资料库、知识文档、交付文档和共享文件空间。

当前平台里的 `document-management` 已经覆盖：

- 文档库与目录树
- 文档上传、下载和预览
- 文档与目录的重命名、移动、删除
- 文档库管理员和所有权管理
- 文档级、目录级权限控制
- 面向系统管理员的后台管理入口

如果你的目标是做“文件中心”而不是“业务记录附件”，文档管理更合适。

### 自定义 UI Schema

适合不改前端代码、但希望把某些页面 Schema 当成可管理资源维护的场景。

`ui-schema-management` 支持把自定义 UI Schema 绑定到路径并启用或停用，适合做页面模板覆盖、运行时页面定制或系统设置类页面治理。

如果你当前更关心的是“标准页面、动态表单和自定义 UI Schema 三条路线该怎么选”，建议继续看 [页面交付：标准页面、动态表单与 UI Schema](./page-delivery-with-dynamic-forms-and-ui-schema)。

## 常见组合方式

### 组合一：业务模型 + 附件上传

这是最常见的业务交付方式。

适合：

- 核心对象比较稳定
- 附件只是记录的补充信息
- 资料不会脱离业务单据独立管理

### 组合二：动态表单 + 文件上传

适合：

- 先快速上线一个采集或登记流程
- 表单结构变化较快
- 需要支持图片、附件或导入文件

这类场景里，动态表单负责采集结构，上传能力负责承接附件。

### 组合三：业务流程 + 文档管理

适合：

- 文件本身需要长期存放和复用
- 文档访问权限和业务单据权限不完全一致
- 同一份资料需要被多人、多角色持续访问

这类场景里，文档管理更像独立模块，而不是单据上的附件栏。

### 组合四：标准能力 + 自定义 UI Schema

适合：

- 主能力仍然留在低代码
- 但某些页面需要单独维护展示方式
- 暂时不想直接进入微前端或自定义前端组件

## 这类需求的推荐推进顺序

1. 先判断对象是不是核心业务模型
2. 再判断文件是“业务附件”还是“独立文档资产”
3. 用动态表单承接轻量采集和快速试点
4. 需要独立资料空间时再引入文档管理
5. 页面模板不合适时，先评估自定义 UI Schema，再决定是否写前端代码

## 什么时候还留在低代码里就够了

- 只是要收集数据、展示列表和维护表单
- 只是要给业务记录补上传、下载和查看附件
- 只是要做一个共享资料库，并配好目录和权限
- 只是要调整某些运行时页面的 Schema

这些需求多数还在平台主线内，优先用现成能力组合更稳。

## 什么时候应该切到融合开发

- 需要自定义文件预览、转码、查毒或外部对象存储策略
- 需要把文档权限扩展到更复杂的主体解析逻辑
- 需要把动态表单和外部系统做更深的程序化同步
- 需要页面交互明显超出平台现有 Schema 和组件边界

遇到这些情况，再进入 [融合开发](../fusion-development/) 会更合适。
