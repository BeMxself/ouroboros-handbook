# 组织、权限、配置与通知

这页适合已经开始做业务模型、页面和流程，但发现“应用已经有骨架了，真正上线还差一组基础能力”的团队。

很多低代码项目并不是卡在表单怎么画，而是卡在下面这些问题：

- 谁属于哪个部门，审批人和接收人怎么找
- 谁能看、谁能改、谁能操作哪些页面和数据
- 哪些值应该部署后还能继续调整
- 业务状态变化后，消息应该通过什么方式触达谁

在潮汐栈里，这些并不是零散功能，而是一组彼此衔接的基础能力。

## 先理解这四类能力各自负责什么

### 组织架构

组织架构负责维护公司、部门和员工关系，是审批流、通知触达和部分权限分配的重要基础数据来源。

它更像“业务身份与组织关系”的底座，而不是登录认证本身。

如果你的项目里会出现部门负责人、员工归属、组织树、审批人按部门选择这类需求，通常都要先把组织数据准备好。

### 用户角色权限

权限能力负责定义“谁能访问什么、谁能执行什么”。

这组能力当前已经覆盖：

- 用户账号维护
- 角色维护
- 角色与用户关系分配
- 页面、接口和数据范围的授权治理

组织架构解决的是“人和组织关系”，权限解决的是“这些人被允许做什么”。两者常常一起出现，但不要混成一件事。

### 配置项

配置项负责承接那些“上线后仍可能继续调整”的系统参数或业务开关。

当前平台里的统一配置能力，不只是简单存一个 key-value，而是围绕配置元数据、值类型、默认值、是否允许修改、是否允许为空、是否过时等规则来治理。

如果一个值可能因环境、客户或阶段不同而变化，就值得优先评估是否抽成配置项。

### 消息通知

通知能力负责把业务事件真正送达到对应的人。

当前平台已经内建统一通知抽象，并支持多渠道接入，包括：

- 站内信
- 微信
- 钉钉
- 邮箱
- 短信
- 电话通知
- 推送类渠道

通知能力并不只是“发一条消息”，它还会和模板、组织结构、流程节点以及页面跳转协同工作。

## 为什么这四类能力通常要一起看

这四类能力经常构成一个完整闭环：

1. 组织架构定义“有哪些人、在哪个组织里”
2. 用户角色权限定义“这些人能做什么”
3. 配置项定义“这个业务在当前环境下按什么参数运行”
4. 消息通知定义“当业务发生变化时，谁会被告知”

如果只做模型和页面，不把这组基础能力补齐，应用往往能演示，但很难稳定交付。

## 一张简单判断表

| 当前问题                           | 优先能力           |
| ---------------------------------- | ------------------ |
| 审批人、负责人、所属部门怎么确定   | 组织架构           |
| 哪些人能看页面、调接口、处理数据   | 用户角色权限       |
| 某个阈值、开关、默认值要不要写死   | 配置项             |
| 状态变化后怎么提醒申请人或处理人   | 消息通知           |
| 上面四类问题在同一个业务里同时出现 | 这四类能力一起规划 |

## 常见组合方式

### 组合一：组织架构 + 权限

适合有明显岗位分工的业务系统。

例如销售、部门主管、财务、管理员分别进入不同页面、看到不同菜单或拥有不同数据范围时，通常都要先把组织和角色关系梳理清楚。

### 组合二：组织架构 + 通知

适合审批、待办、异常提醒类场景。

这类场景通常先依赖组织结构找到负责人、申请人或相关员工，再通过通知模板把消息发出去。

### 组合三：配置项 + 通知

适合“上线后仍要逐步调节”的业务能力。

例如先只打开站内提醒，再逐步打开短信或企业协同渠道；或者通过配置控制某项提醒是否启用、阈值是多少、走哪个模板。

### 组合四：组织 + 权限 + 配置 + 通知

这是最接近真实业务交付的一组组合。

它意味着你的应用已经不仅能录入和展示数据，还开始具备组织协作、访问控制、运行参数治理和消息触达能力。

## 推荐推进顺序

如果你是在做一个新的业务闭环，通常建议按下面顺序推进：

1. 先准备组织数据，明确公司、部门、员工和负责人关系
2. 再准备系统用户与角色，把关键岗位纳入权限体系
3. 把核心页面、接口和数据范围权限先配成最小闭环
4. 把容易变化的阈值、开关、默认值抽成配置项
5. 为审批、待办、状态变化等关键事件补通知模板
6. 用一个真实业务流程把“发起、处理、通知、回看”走通

这样做的好处是，后面无论继续留在低代码，还是切到融合开发，边界都会更清楚。

## 什么时候这组能力留在低代码里就够了

- 组织结构比较标准，不需要自定义复杂的人岗解析逻辑
- 权限主要是角色、页面、接口和基础数据范围控制
- 配置项主要是参数、开关、默认值，不需要额外做复杂配置中心整合
- 通知主要是模板配置、渠道接入和流程触发，不需要自建分发编排系统

这类场景里，优先复用平台现成能力通常更稳，也更容易维护。

## 什么时候应该切到融合开发

- 组织身份需要和外部主数据、人事系统做更深度同步
- 权限模型需要自定义主体类型、特殊授权规则或更复杂的数据域控制
- 配置项需要和外部配置中心、发布体系或灰度策略深度打通
- 通知需要自定义渠道适配器、复杂分发策略或更特殊的模板处理

这时更适合进入 [融合开发](../fusion-development/)，把扩展点和标准能力组合起来用。
