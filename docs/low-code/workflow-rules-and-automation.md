# 流程、规则与自动化

这页适合已经搭好了业务模型和基础页面，下一步准备把“业务怎么流转、规则怎么判断、系统怎么自动处理”补完整的团队。

很多低代码项目做到这里，最容易混淆的不是功能有没有，而是这些能力到底该怎么分工：

- 什么时候该上审批流
- 什么时候该上逻辑流
- 一条规则该写成表达式还是脚本
- 通知应该算流程的一部分，还是单独补

在潮汐栈里，这几类能力可以组合，但不应该混用。

## 先理解这几类能力各自负责什么

### 审批流

审批流解决的是“人要不要参与决策和处理”。

它适合承接：

- 申请、审核、驳回、转办、会签
- 候选人解析和待办分发
- 审批中心处理和统计回看

如果你的业务关键点在“谁来审批、什么时候批、批完进入下一步”，优先看审批流。

### 逻辑流

逻辑流解决的是“系统如何按节点顺序自动处理一段业务逻辑”。

它适合承接：

- 数据映射、校验、分支判断
- 查询、插入、更新等数据处理
- 把多个步骤组织成一条可见、可调的处理链

如果你的核心问题是“系统收到输入后要怎么自动执行一串处理”，优先看逻辑流。

### 表达式

表达式适合承接短小、清晰、低副作用的动态规则。

它通常更适合：

- 一个字段的默认值
- 一条简单条件判断
- 少量取值、拼接和格式转换

如果一条规则应该做到“看一眼就能读懂”，表达式通常是第一选择。

### 脚本

脚本适合承接比表达式更复杂、但还没必要升成完整流程或代码扩展的逻辑。

它通常更适合：

- 多步计算
- 中间变量处理
- 更复杂的分支和循环
- 在脚本节点里做局部规则处理

如果规则已经不适合写成一行表达式，但仍然想保留较强灵活性，脚本会更合适。

### 通知

通知解决的是“业务动作发生后，谁会被告知，以及怎么被告知”。

它通常用于：

- 审批待办提醒
- 状态变更通知
- 异常告警
- 处理结果回传

通知本身不负责推进业务逻辑，但它几乎总会和审批、流程、规则一起出现。

## 一个简单判断表

| 当前问题                               | 优先能力               |
| -------------------------------------- | ---------------------- |
| 需要人工审批、待办分发和结果处理       | 审批流                 |
| 需要系统自动执行一串处理步骤           | 逻辑流                 |
| 只是做一条很短的取值或条件判断         | 表达式                 |
| 规则已经比表达式复杂，但还没到流程编排 | 脚本                   |
| 动作发生后要提醒申请人、审批人或处理人 | 通知                   |
| 既有人工处理，又有系统自动后续动作     | 审批流 + 逻辑流 + 通知 |
| 流程里的局部节点需要短规则或复杂规则   | 逻辑流 + 表达式 / 脚本 |

## 为什么这些能力通常要一起看

它们在真实业务里经常形成这样一条链路：

1. 用表达式或脚本承接局部规则
2. 用逻辑流把自动处理过程串起来
3. 需要人工决策时进入审批流
4. 在关键节点用通知把结果推给相关人

也就是说：

- 表达式和脚本更像“规则载体”
- 逻辑流更像“系统处理编排”
- 审批流更像“人工处理编排”
- 通知更像“流程结果触达”

把这四层关系理顺后，很多场景就不会一上来陷入“是不是都用脚本写掉”或者“是不是所有流转都塞进审批流”。

## 常见组合方式

### 组合一：审批流 + 通知

适合标准申请审批类场景。

审批流负责发起、流转、待办和结果，通知负责把待办、通过、驳回等状态及时发给对应人员。

### 组合二：逻辑流 + 表达式

适合大多数标准自动化场景。

逻辑流负责主处理链，表达式负责节点里的轻量规则判断。这通常是最容易维护的一种组合。

### 组合三：逻辑流 + 脚本

适合处理链里存在局部复杂规则的场景。

整体流程仍然留在可视化编排里，复杂但局部的规则落到脚本节点，不必过早进入代码扩展。

### 组合四：审批流 + 逻辑流 + 通知

适合“人工决策 + 系统自动后处理”的业务闭环。

例如请假、采购、合同、报销、变更申请这类场景，常常由审批流承接人工环节，再由逻辑流做状态更新、数据写入、后续分发或回写动作，最后再补通知。

## 推荐推进顺序

如果你在做一个新的业务闭环，通常建议按下面顺序推进：

1. 先判断这一步是“人来决定”还是“系统自动处理”
2. 规则足够短时先用表达式，不要一开始就上脚本
3. 规则开始变长时，再升级为脚本
4. 需要把多个自动步骤串起来时，再建立逻辑流
5. 需要人工审批节点时，再引入审批流
6. 为待办、结果和异常节点补通知模板

这样推进的好处是，每一层能力都只解决自己最擅长的问题。

## 一个很实用的边界原则

### 审批流不是通用自动化引擎

审批流最适合承接“人工审批语义”，不适合把所有系统自动处理都塞进去。

### 逻辑流不是所有规则的最终归宿

如果只是很短的取值或条件判断，直接用表达式会更轻；如果只是局部复杂计算，用脚本也可能更省。

### 表达式和脚本也不该承担长期失控的业务复杂度

当一段规则已经影响多人协作、重复复用很多次，或者明显超出配置可维护范围时，就该考虑进入 [融合开发](../fusion-development/)。

## 什么时候这组能力留在低代码里就够了

- 规则主要是标准业务判断和数据处理
- 自动处理链可以被现有节点、表达式和脚本承接
- 审批流程仍然是标准候选人、节点流转和待办处理
- 通知主要是模板配置和渠道选择

这类场景通常仍然适合留在低代码主线内完成。

## 什么时候应该切到融合开发

- 需要自定义逻辑流节点或新的执行能力
- 需要扩展审批候选人解析、任务动作或引擎适配
- 需要补充新的表达式上下文、脚本上下文或规则能力
- 需要通知渠道适配器、特殊分发策略或更深的外部系统联动

这时更适合沿着平台扩展点进入融合开发，而不是继续把复杂度堆在配置里。
