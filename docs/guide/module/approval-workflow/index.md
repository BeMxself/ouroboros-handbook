# 审批流

审批流专题主要覆盖“定义流程、发起流程、处理流程、查看统计”四类操作。

2. [从需求到交付](../../../low-code/from-requirement-to-delivery)
3. [流程、规则与自动化](../../../low-code/workflow-rules-and-automation)
4. [审批流模型](../../../concepts/approval-workflow-model/)

这组页面解决的是“审批流在整条交付链里应该什么时候用、要先准备什么、它和逻辑流/通知/组织数据怎么协同”；而本目录更适合在你已经确认要做审批闭环之后，再来查页面操作和管理入口。

## 先理解审批流适合解决什么

审批流最适合承接“人要参与决策和处理”的长流程，例如：

- 请假、报销、采购、合同、付款、变更申请
- 需要按岗位、角色或负责人逐步审批
- 需要待办、流转记录、审批意见和过程状态
- 需要区分发起人、审批人、管理员和统计查看者

如果你的需求本质上只是“系统自动处理一段逻辑”，通常不应该先上审批流，而应该优先看 [逻辑流](../../tutorial/logicflow/) 或 [流程、规则与自动化](../../../low-code/workflow-rules-and-automation)。

换句话说：

- 审批流：更偏“人工处理编排”
- 逻辑流：更偏“机器执行处理”
- 通知：更偏“把结果和待办触达到人”

这三者经常组合，但不应该混成同一件事。

## 开始前最好先准备什么

如果你当前是在搭第一个审批场景，建议先把下面四件事准备清楚，再进入具体页面：

1. 谁来设计流程
2. 谁来发起流程
3. 谁来审批流程
4. 谁来看统计和处理异常实例

除此之外，通常还要把三类基础能力先补齐：

1. 组织架构：至少能找到员工、部门、岗位或负责人
2. 表单或业务数据：审批时要展示和填写什么
3. 消息与权限：待办怎么触达、谁能发起、谁能处理

如果这些前置条件还没准备好，后面最容易反复返工的就是候选人、字段权限和流程分支。

## 适用角色

- 管理员或流程设计人员：维护审批流定义、部署和下线
- 业务申请人：发起审批
- 审批人员：在审批中心处理流程
- 管理人员或支持人员：查看统计、排查实例和处理异常流程

## 先理解这几页的分工

- [审批流管理](./approval-management/)：用于设计、部署和维护审批流定义
- [发起审批](./approval-start/)：用于业务申请人发起具体审批请求
- [审批中心](./approval-center/)：用于审批人查看和处理待办任务
- [审批流统计](./approval-statistics/)：用于管理视角查看执行情况和处理异常实例

也就是说，本目录不是只有“审批人点同意/拒绝”这一层，而是覆盖了设计时、发起时、处理时和管理时四类入口。

## 常见组合关系

审批流很少单独使用，真实项目里通常会和下面几类能力一起出现：

- 和 [组织架构管理](../organization/) 组合：解决“审批人从哪里找”
- 和 [用户角色权限](../rbac/) 组合：解决“谁能发起、谁能进入哪些页面”
- 和 [消息通知](../notification/) 或 [通知模板](../../tutorial/notification/) 组合：解决“待办、通过、驳回怎么触达”
- 和 [逻辑流](../../tutorial/logicflow/) 组合：解决“审批通过后系统还要自动做什么”

如果你在这里已经开始关心“审批后要自动回写数据、发消息、更新状态”，建议同时回看 [流程、规则与自动化](../../../low-code/workflow-rules-and-automation)。

## 使用建议

- 先完成组织架构和角色准备，再设计审批流，通常更容易配置审批人
- 先用一个最小场景把“发起到审批完成”的闭环走通，再逐步增加分支、会签和定时能力
- 如果你是为了完成一个业务闭环来这里，建议结合 [从需求到交付](../../../low-code/from-requirement-to-delivery) 一起阅读
- 如果你还在判断“是不是该上审批流”，先回到 [流程、规则与自动化](../../../low-code/workflow-rules-and-automation)
