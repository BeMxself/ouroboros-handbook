# 创建第一个应用

适用对象：已经能进入平台，希望完成第一次业务开发的读者。

## 这一步到底要完成什么

不要一开始就把“完整项目”当作目标。第一次上手更合适的目标是先搭一个最小应用骨架，让你后面做页面、流程、权限时都有稳定落点。

通常这一步只需要做到三件事：

- 创建一个应用
- 打开应用并进入开发入口
- 建立一个最小业务模型骨架

做到这里，就足够进入下一步业务功能串联了。

## 开始前先确认

在进入这一步之前，最好已经满足下面两个前提：

1. 你已经能正常进入开发平台
2. 你已经大致知道自己要练习哪个业务对象，例如“请假单”“客户”“设备台账”

如果还没有想好业务对象，也没关系。第一次练习时，优先选下面这类简单对象会更稳：

- 字段数量不多
- 关系不复杂
- 不需要一开始就做多角色协同

## 推荐范围

第一次创建应用时，建议把范围控制在“一个应用 + 一个核心业务对象”。

推荐例子：

- 请假管理
- 客户档案
- 设备台账
- 合同登记

不建议第一次就把下面这些一起做进来：

- 多模块、多子系统
- 大量审批分支
- 自定义前端扩展
- 外部系统集成

这些能力后面都能补，但不适合作为第一次创建应用时的起点。

## 推荐步骤

### 1. 先创建应用骨架

先到 [开发应用](../guide/tutorial/apps/) 中完成应用创建，把应用名、应用标题和基础配置定下来。

这一层解决的是：

- 这个业务空间叫什么
- 后续模型、页面和菜单挂到哪里
- 团队后续进入哪个应用继续开发

### 2. 打开应用，确认开发入口

创建完成后，进入应用，确认你已经能找到：

- 业务模型入口
- 应用配置入口
- 后续页面和模块相关入口

这一小步的目标不是继续深配，而是确认“应用骨架已经可进入、可继续开发”。

### 3. 建立一个最小业务模型

接着看 [业务模型](../guide/tutorial/business-model/) 和 [开发业务模型](../guide/tutorial/business-model/develop-business-model/)，先为你的第一个业务对象建立最小模型。

最开始只需要先定清楚：

- 模型名
- 模型标题
- 2 到 5 个核心字段

如果需要理解模型、字段和关系的抽象，再补看 [核心概念](../concepts/)。

### 4. 先不要急着做复杂功能

这一阶段先不急着做：

- 审批流
- 复杂权限
- 消息通知
- 自定义前端
- 自定义后端扩展

第一次上手时，先把“应用存在、模型存在、入口清楚”做出来，后面串闭环会轻松很多。

## 做到什么算完成

完成这一阶段时，你应该至少能做到：

- 在平台中创建并打开一个应用
- 找到业务模型相关入口
- 建立一个最小业务模型骨架
- 知道后续该往页面、流程、权限哪条线继续走

如果你已经能做到这些，就说明“第一个应用”已经建好了。

## 常见误区

### 一开始就想把完整业务系统搭完

这通常会让你在应用、模型、流程、权限几条线之间反复切换，反而更难入门。

### 应用刚建好，就马上开始前后端扩展

第一次练习时，优先先把低代码主线跑起来。只有在平台标准能力明显不够时，再进入融合开发。

### 还没想清楚业务对象，就先建很多应用

第一次上手，应用数量越少越好。先用一个应用把完整路径跑通，收益最大。
