# 完成第一个业务功能

适用对象：已经创建了应用，准备串起一个最小业务功能的读者。

## 这一步的目标

这一步不是把应用“继续做大”，而是把它真正跑成一个最小业务闭环。

一个最小业务闭环，通常至少要满足：

- 能录入或创建数据
- 能查看、检索或修改数据
- 关键业务动作能被系统处理
- 不同角色知道自己该从哪里进入

也就是说，这一步的重点是“把应用从骨架变成能跑通的一条业务链”。

## 推荐练习对象

建议仍然选择一个简单业务对象，例如：

- 请假单
- 客户信息
- 设备台账
- 合同登记

推荐优先选择那种：

- 字段结构清楚
- 至少有一个创建动作
- 至少有一个查看或处理动作
- 不必一开始就接第三方系统

## 推荐闭环步骤

### 1. 把业务模型补到可用

先在 [开发业务模型](../guide/tutorial/business-model/develop-business-model/) 中把模型补到可实际使用。

这一阶段至少要补齐：

- 核心字段
- 基础显示字段
- 必要的关联关系

如果模型本身还不稳定，后面的页面和流程通常都会反复返工。

### 2. 让页面能录、能看、能查

接着确认这个模型至少已经能支撑：

- 新建
- 编辑
- 查看详情
- 列表检索

如果当前页面仍然主要是标准列表、表单和详情页，优先继续留在低代码主线即可，不必急着进入前端扩展。

### 3. 给关键动作补上系统处理

接下来要判断，这个业务闭环里的“处理动作”到底更适合哪一种：

- 只是系统自动处理：优先看 [逻辑流](../guide/tutorial/logicflow/)
- 需要人工发起、审批、流转：优先看 [审批流](../guide/module/approval-workflow/)
- 只是简单状态变化或基础规则：可以先停留在低代码页面和标准能力内

第一次练习时，不需要把逻辑流和审批流都接上，只要选一条最贴近业务本质的方式即可。

### 4. 补上最小身份与访问边界

即使是第一个业务功能，也建议至少补最小身份与访问边界。

通常要先确认：

- 谁能看到入口
- 谁能创建数据
- 谁能处理或审批数据
- 谁会因为主体/组织关系拿到不同的数据范围

相关页面建议配合看：

- [组织、权限、配置与通知](../low-code/organization-permissions-configuration-and-notification)
- [权限与访问控制](../guide/advance/authority-and-access-control)

### 5. 做一次完整演练

最后至少做一次完整闭环验证：

1. 创建一条业务数据
2. 进入列表或详情查看结果
3. 执行一次关键处理动作
4. 确认权限和入口没有明显错位

第一次练习时，这一步比继续加新功能更重要。

## 一个最低可用完成标准

如果满足下面这些条件，通常就可以算“第一个业务功能已经跑通”：

- 业务数据能创建
- 页面能查看和检索
- 至少一个关键业务动作已经被系统承接
- 至少一类角色权限已经区分开
- 你能从头到尾演练一遍

## 常见误区

### 同时上很多业务对象

第一次练习时，业务对象越多，越容易把问题扩散到模型关系、权限和流程配置上。

### 一开始就做复杂审批和复杂权限

先把最小闭环走通，再慢慢加细粒度控制，通常更稳。

### 页面刚有雏形，就急着做代码扩展

如果当前页面仍然能由标准列表、表单和流程承接，先不要过早切到融合开发。

## 下一步

如果你已经能完成一个最小业务闭环，可以继续根据角色进入：

- 想继续扩展更多业务交付主题：看 [低代码开发](../low-code/)
- 想判断哪些部分该切到代码扩展：看 [融合开发](../fusion-development/)
- 想回看低代码主线路径：看 [低代码开发总览](../low-code/overview)
