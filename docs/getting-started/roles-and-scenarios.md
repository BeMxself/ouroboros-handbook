# 角色与典型场景

在潮汐栈里，不同角色看到的平台价值并不完全一样。先找准你的角色，后面的阅读路径会顺很多。

但这些角色背后有一个共同目标：不是只把系统“先做出来”，而是要在需求持续变化、系统需要对接、应用需要长期维护的情况下，依然保持较高的交付效率和较低的整体成本。

## 首次接触平台的人

典型目标：

- 先理解平台能做什么
- 尽快把开发平台跑起来
- 完成一个最小可用示例
- 判断这个平台是否适合长期承接真实项目，而不只是快速做原型

建议阅读：

1. [平台介绍](../overview)
2. [环境准备](./environment-setup)
3. [安装开发平台](./install-develop-platform)

## 业务开发人员

典型目标：

- 创建应用
- 定义业务模型和页面
- 配置流程、权限、通知等业务能力
- 在需求多次调整时，尽量继续通过模型和配置完成迭代

建议阅读：

1. [低代码开发](../low-code/)
2. [低代码开发总览](../low-code/overview)
3. [从需求到交付](../low-code/from-requirement-to-delivery)
4. [业务模型](../guide/tutorial/business-model/)

## 专业开发人员

典型目标：

- 判断哪些需求适合低代码，哪些需要写代码
- 通过前后端扩展承接复杂需求
- 保持与低代码模型协同
- 避免因为单点复杂需求而把整套低代码交付链推倒重来

建议阅读：

1. [融合开发](../fusion-development/)
2. [融合开发总览](../fusion-development/overview)
3. [何时用低代码，何时写代码](../fusion-development/when-to-use-code)
4. [平台扩展点总表](../fusion-development/extension-points)
5. [高低开协同模式](../fusion-development/high-low-code-collaboration)

## 实施与运维人员

典型目标：

- 安装与初始化平台
- 部署开发出来的应用
- 负责升级、维护和排障
- 让平台既能支持个人试用和团队协作，也能逐步走向统一运维

建议阅读：

2. [运维总览](../operations/overview)
3. [部署架构](../operations/deployment-architecture)
