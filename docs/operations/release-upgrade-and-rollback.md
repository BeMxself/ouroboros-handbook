# 升级、回滚与维护

平台一旦进入团队使用阶段，升级和维护方式就必须提前设计，而不是等到出问题时再补。

## 这部分要解决什么问题

- 平台版本升级的窗口和流程
- 应用发布与平台升级的边界
- 回滚所需的数据和配置保留策略
- 升级前后的验证清单

## 升级前先建立的基本原则

### 先分清变更类型

升级前先判断这次变更属于哪一类：

- 平台版本升级
- 平台配置调整
- 依赖组件升级
- 业务应用发布

如果四类动作混在同一个窗口执行，后续出现问题时会很难判断根因，也会让回滚边界变得模糊。

### 不要跳过可回退准备

每次升级前，都应该先回答三个问题：

1. 这次变更失败后，能回退到哪个状态
2. 需要保留哪些数据库、配置和运行镜像
3. 回滚由谁执行，多久内必须完成

### 先演练再进入长期环境

如果这套升级路径是第一次执行，建议至少先在体验环境或团队共享环境演练一次，确认：

- 初始化脚本和升级动作是可重复的
- 配置差异是可记录的
- 升级后验证项是明确的

## 升级前检查清单

执行升级前，建议逐项确认：

- 当前平台版本、目标版本和变更范围
- 当前环境的关键配置已经留档
- 数据库已完成备份，并验证了恢复方式
- 消息、文件、镜像或部署清单已保留必要副本
- 业务应用当前运行状态稳定，没有待处理故障
- 升级窗口、影响范围和回滚负责人已明确
- 升级后的验证清单已经准备好

## 推荐升级流程

### 1. 冻结变更范围

在升级窗口前，先暂停与平台运行方式强相关的额外变更，避免一边升级平台、一边改配置、一边发应用。

### 2. 做备份和留档

至少保留以下内容：

- 数据库备份
- 平台配置和环境变量
- 部署脚本、镜像标签或发布清单
- 与登录、存储、消息相关的关键外部配置

### 3. 先在低风险环境验证

如果升级涉及版本跨度较大、依赖组件变更或运行方式调整，建议先在体验环境或共享环境验证：

- 能否成功启动
- 数据是否正常加载
- 应用管理和核心开发入口是否可用
- 业务应用是否还能正常发布和运行

### 4. 正式执行升级

正式升级时，建议把动作控制在可记录的顺序里：

1. 记录当前状态
2. 执行平台和依赖的目标变更
3. 检查平台服务是否恢复
4. 检查关键依赖是否恢复
5. 执行升级后验证

### 5. 保留观察窗口

升级完成不代表结束。建议在一段观察窗口内持续关注：

- 平台登录与核心入口是否稳定
- 数据库和消息连接是否持续正常
- 应用发布、运行和日志是否出现新的异常

## 回滚策略

### 什么情况应该立即回滚

如果出现以下情况，通常不要继续“边试边修”，而应优先回到已知稳定状态：

- 平台无法启动或无法登录
- 核心依赖连接持续失败
- 业务应用大面积不可用
- 升级后出现无法快速定位的数据或配置问题

### 回滚前先确认边界

回滚前先明确：

- 回滚的是平台版本、依赖版本，还是配置
- 数据是否已经发生不可逆变更
- 业务应用是否也需要回到之前版本

### 推荐保留的回滚资产

- 升级前数据库备份
- 升级前平台配置快照
- 升级前镜像、包或部署清单
- 升级前后的执行记录和验证结果

## 升级后验证清单

升级完成后，建议至少验证下面这些项目：

- 登录页和开发平台主页可访问
- 应用管理、模型管理等关键入口可进入
- 数据库、RabbitMQ、Docker API 等关键依赖无明显报错
- 已有业务应用可正常打开或运行
- 新建、发布或更新一个最小变更能够正常完成
- 平台日志中没有持续增长的严重错误

## 日常维护建议

除了版本升级，长期维护还应形成固定节奏：

- 定期核对平台版本和环境差异
- 定期检查备份是否真实可恢复
- 定期清理无效镜像、容器和过期运行数据
- 定期回顾故障记录，沉淀团队内部操作手册

## 需要专项方案的场景

如果你的环境具备以下特征，建议单独形成团队方案，而不要只依赖通用安装步骤：

- 多环境并行维护
- 严格的变更审批或发布窗口
- 外部统一认证、对象存储或消息系统深度接入
- 业务应用数量较多，且发布节奏频繁
