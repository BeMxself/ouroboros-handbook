# 部署架构

在运维视角下，潮汐栈的部署对象通常至少包含三类内容：

- 平台本身
- 平台依赖的基础组件
- 通过平台开发和交付的业务应用

这背后对应的是一种很明确的设计取向：平台既要能支撑个人或小团队快速启动，也要能逐步走向团队协作和统一运维，而不是只能适配单一规模、单一部署形态。

## 先分清三层边界

### 平台层

主要指开发平台本身，以及它对登录、模型管理、应用发布和运行控制的支撑能力。

### 依赖层

主要指数据库、消息中间件、Docker 运行环境，以及文件存储等基础设施。它们通常需要独立持久化、监控和备份。

### 应用层

主要指通过潮汐栈开发并发布出来的业务应用。应用的发布节奏、灰度方式和故障处理，不一定和平台升级绑定。

## 常见环境形态

### 本地体验环境

适合个人学习、功能验证和最小流程演示，特点是：

- 部署快
- 依赖组件可以合并在同一台机器
- 允许一定程度的手工操作
- 更看重开箱即用和应用可迁移性

### 团队共享环境

适合业务开发、测试联调和培训演示，特点是：

- 需要稳定的访问地址
- 需要团队共用的账号、数据和日志入口
- 应用发布和平台维护开始出现明确分工
- 通常会在同一平台上协作多个项目或多个客户应用

### 长期运行环境

适合持续承载业务应用的正式环境或类生产环境，特点是：

- 依赖组件必须持久化
- 平台升级和应用发布要有变更窗口
- 必须具备备份、监控、恢复和回滚能力
- 往往会开始引入统一账号、统一认证和平台级治理能力

## 这三类环境背后的组织场景

如果从组织方式看，潮汐栈通常要兼顾三种典型场景：

1. 项目团队成员各自在本地启动平台，分别开发不同项目
2. 项目团队在共享平台上协同开发和管理多个项目
3. 企业 IT 团队把平台作为统一应用底座来管理一组内部系统

这也是为什么部署架构不能只考虑“能不能启动”，还要考虑应用迁移、多人协作、统一身份管理和平台级能力边界。

## 部署前要确认的关键问题

- 这是本地体验环境，还是团队长期环境
- 数据、消息和文件如何持久化
- 平台服务和业务应用是否部署在同一运行环境中
- 外部访问入口、域名和网络边界如何控制
- 谁负责升级，谁负责回滚
- 应用发布与平台升级是否分离

## 推荐的最小架构认知

在没有复杂基础设施要求时，可以先按下面的责任划分理解整套环境：

1. 一组平台服务，负责平台管理和应用运行控制
2. 一组基础依赖，负责数据库、消息和运行时能力
3. 一组业务应用运行容器，负责承载实际交付的业务系统

这样做的好处是：

- 平台问题、依赖问题和应用问题更容易分层定位
- 平台升级时，不必默认同时重做应用发布
- 备份和恢复时，知道哪些数据属于平台，哪些属于业务
- 环境可以先从单机多应用起步，再逐步拆分为更稳定的长期运行形态

## 平台升级和应用发布的关系

建议从一开始就把这两件事区分开：

- 平台升级关注平台版本、运行方式、依赖兼容性和配置变更
- 应用发布关注业务变更、模型发布、页面配置和业务验证

如果团队把这两类动作混在一起执行，后续排障和回滚都会明显变难。
