# 任务、消息与实时链路维护

这页适合已经在正式环境里启用了调度任务、消息队列或 WebSocket 的实施与运维团队。

很多平台一开始只把这几类能力当成“某个业务功能的一部分”，但一旦它们真正进到生产环境，就会变成三条必须长期维护的运行链路：

- 任务链路
- 消息链路
- 实时连接链路

它们和普通页面访问不一样，往往不是“能打开页面就代表没问题”。

## 为什么这三类能力值得单独维护

因为它们分别会带来不同的运行风险：

- 调度任务会影响定时报表、同步、巡检和提醒
- 消息队列会影响异步消费、事件分发和上下游解耦
- WebSocket 会影响在线状态推送、进度反馈和实时通知

如果只盯着平台首页是否可访问，通常发现不了这些链路已经部分失效。

## 先把三条链路拆开看

### 调度链路

从当前平台实现看，调度能力大致由下面几段组成：

1. 任务定义被发现和装载
2. Quartz 负责触发时机
3. 调度任务桥接到 LogicFlow 或对应执行逻辑
4. 执行结果继续进入业务处理链

运维真正要关心的不是“有没有任务功能”，而是：

- 任务有没有被正确装载
- 触发时间和时区是不是符合预期
- 触发后是否真的进入了执行链
- 升级、回滚或配置调整后任务是否继续有效

### 消息链路

当前平台里的消息能力是“统一 MQ 抽象 + RabbitMQ 默认适配 + 运行时绑定治理”。

从运维视角看，这条链路至少包含：

1. 绑定定义是否被正确加载
2. `MQBridge` 是否完成初始化
3. RabbitMQ 是否可用
4. 生产者和消费者是否正常工作
5. 消息是否真正进入后续处理链

这意味着消息链路的故障，不一定只发生在 RabbitMQ 本身，也可能出现在绑定加载、消费者装配或平台升级后的运行时接入环节。

### 实时连接链路

WebSocket 运行时更像一条“长连接 + 鉴权 + 路由”的复合链路。

从当前能力边界看，至少要经过：

1. 前端连接入口可达
2. 连接鉴权成功
3. 会话和订阅建立成功
4. 消息按广播、用户级或会话级语义路由
5. 前端真实收到推送

所以实时链路的问题，也不一定等于“WebSocket 服务挂了”，它还可能是网关、鉴权、会话或订阅路径出了问题。

## 一张简单判断表

| 现象                             | 优先检查哪条链路   |
| -------------------------------- | ------------------ |
| 定时任务没跑、周期提醒没发       | 调度链路           |
| 异步处理不再推进、事件迟迟不消费 | 消息链路           |
| 页面在线但状态不刷新、推送收不到 | 实时连接链路       |
| 后台触发成功，但前端没有及时变化 | 消息链路或实时链路 |
| 升级后多个自动化能力同时异常     | 三条链路一起复核   |

## 三类链路各自重点看什么

### 调度链路的日常关注点

- 关键任务是否已装载
- Cron 或一次性触发时间是否符合预期
- 时区和运行环境时间是否一致
- 触发后对应流程是否执行成功
- 平台重启或升级后任务是否恢复

如果一个任务“看起来还在”，但业务结果没有落地，问题往往不只在调度触发本身，还要继续往执行链里查。

### 消息链路的日常关注点

- RabbitMQ 连接是否稳定
- 关键绑定是否正常加载
- 消费者是否已经装配并持续运行
- 是否出现持续消费失败或明显堆积
- 平台变更后是否还存在生产和消费断链

如果某个业务动作已经提交成功，但后续异步动作完全没发生，优先把消息链路当成独立故障域来看。

### 实时连接链路的日常关注点

- WebSocket 入口是否可达
- 反向代理、网关和升级策略是否影响连接
- 鉴权链路是否与当前认证配置一致
- 用户级、会话级消息是否还能按预期送达
- 连接异常、断连重连或订阅失败是否显著增加

如果系统功能都正常，但用户持续反馈“页面不刷新”“状态不更新”，大概率不是业务逻辑本身的问题，而是实时链路没有闭环。

## 变更前后要额外复核什么

只要涉及下面这些变化，建议都把这三条链路当成专项检查项：

- 平台升级
- 认证方式调整
- RabbitMQ 或数据库连接变化
- 网关、反向代理或端口策略变化
- 时区、运行节点或部署架构变化

最少建议复核：

1. 一个关键调度任务能否继续按时触发
2. 一条关键消息链路能否继续生产和消费
3. 一个真实在线页面能否继续收到推送

## 常见误判

### 误判一：平台能打开，就说明自动化能力没问题

不成立。

调度、消息和 WebSocket 都可能在“页面看起来正常”的情况下单独失效。

### 误判二：RabbitMQ 正常，就说明消息链路没问题

不成立。

绑定、消费者装配、平台运行时接入都可能独立出问题。

### 误判三：用户收不到推送，就是前端问题

不一定。

入口、鉴权、订阅、会话路由和发送链路任何一段出错，最终都会表现成“前端没收到”。

## 一个实用的维护原则

把这三类能力都当成“独立运行链路”维护，而不是把它们藏在业务功能后面。

这样做的好处是：

- 巡检更有针对性
- 升级验证更明确
- 故障定位更快
- 团队分工更清楚
