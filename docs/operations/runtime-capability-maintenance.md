# 运行能力维护要点

当平台开始真正承接文件、文档、调度任务、消息链路和实时推送后，运维工作就不再只是“应用能不能启动”，而是要保证这些运行能力长期稳定。

这页关注的是几个容易在实施阶段被低估的方向：

- 文件和文档存储怎么维护
- 调度和消息能力怎么守住稳定性
- 实时连接和业务日志怎么纳入日常运维

## 先看哪些能力已经会影响部署方案

### 文件上传

文件上传能力默认就会带来本地存储、访问地址和清理策略问题。

当前 `ouroboros-file-upload` 已经包含：

- 默认文件系统存储实现
- 文件访问地址生成
- 临时文件与分片文件清理

运维上至少要明确：

- 上传文件落在哪类存储上
- 存储目录是否持久化
- 临时文件清理是否符合环境要求
- 备份是否覆盖上传目录

### 文档管理

文档管理是独立的资料存储能力，不应只按普通附件思路维护。

当前 `document-management` 默认支持：

- 独立存储根路径配置
- 文件系统存储
- 配额、启停和后台管理
- 文档库、目录和权限数据持久化

实施时至少要确认：

- `ouroboros.document-management.storage.filesystem.base-path` 是否落在持久化磁盘
- 文档库元数据与实际文件是否一起备份
- 文档容量和权限管理是否有人负责

### 调度任务

调度能力一旦承接业务动作，就会直接影响定时报表、同步任务、提醒和巡检。

当前平台里的调度能力已经把任务发现、Quartz 调度和 LogicFlow 执行桥接串起来，所以运维上至少要关注：

- 任务是否真的被装载
- 触发时间和时区是否符合预期
- 调度触发后对应流程是否正常执行
- 升级后任务是否继续可用

如果你准备把调度任务、消息队列和 WebSocket 拆成独立运维对象治理，可以继续看 [任务、消息与实时链路维护](./task-message-and-realtime-link-maintenance)。

### 消息队列

消息队列能力会把平台稳定性延伸到中间件侧。

当前平台默认 RabbitMQ 适配已经接入消息绑定、生产消费和 LogicFlow 消费桥接，所以运维上至少要关注：

- RabbitMQ 是否可用
- 绑定和消费者是否正常装载
- 发布和消费链路是否持续报错
- 平台升级或配置调整后消息链路是否还通

### WebSocket

WebSocket 能力会把“平台可用”扩展为“连接和推送是否可用”。

当前运行时已经包含连接鉴权、会话管理和消息路由，因此运维上至少要关注：

- 连接入口是否可达
- 鉴权相关配置是否一致
- 会话消息和用户定向消息是否还能正常送达

### 业务日志

`ouroboros-business-log` 可以把业务操作沉淀成统一日志记录，这对审计、回溯和排障都很关键。

如果团队已经启用了业务日志，建议把它纳入正式运维范围，而不是只在出问题时临时查看。

如果你要进一步梳理业务日志和审计模板该怎么配合使用，可以继续看 [业务日志与审计回溯](./business-logs-and-audit-traceability)。

## 备份时不要漏掉什么

除了数据库和部署脚本，建议把下面几类内容一起纳入备份范围：

- 上传文件目录
- 文档管理存储目录
- 平台与应用配置
- 任务、消息和文档相关的运行配置
- 业务日志相关的持久化数据

如果只备份数据库而不备份文件和配置，很多恢复场景仍然无法真正完成。

## 日常巡检建议重点看什么

### 存储侧

- 上传目录和文档存储目录是否持续增长
- 挂载和持久化磁盘是否正常
- 容量是否接近风险阈值

### 任务侧

- 关键任务是否按预期触发
- 是否出现长时间不再执行或持续失败的任务

### 消息侧

- RabbitMQ 是否健康
- 关键消费链路是否堆积或异常

### 实时侧

- WebSocket 连接是否稳定
- 鉴权失败或连接异常是否显著增加

### 审计与排障侧

- 业务日志是否还在持续写入
- 平台服务日志、业务日志和依赖日志能否串起来排查

## 升级和变更时的额外检查项

当你准备升级平台、切换部署方式或启用这些能力时，建议额外检查：

1. 存储路径和挂载是否变化
2. RabbitMQ、数据库等依赖连接是否变化
3. 调度任务和消息绑定是否需要重新验证
4. WebSocket 入口和鉴权链路是否被代理或网关配置影响
5. 备份和回滚方案是否覆盖新增能力

## 一个简单原则

只要平台开始承接文件、任务、消息和实时连接，这些能力就都应该进入正式运维范围，而不是被当成“业务层的小功能”。
