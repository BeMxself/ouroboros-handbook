# 备份、监控与排障

运维工作进入稳定期后，最重要的是可观测性和恢复能力。

## 最低保障能力

- 关键数据和配置的备份
- 平台运行状态与日志的收集
- 常见故障的排查路径
- 出问题后的恢复和回滚手段

## 备份时至少覆盖什么

建议至少覆盖以下几类内容：

- 数据库数据
- 平台配置、环境变量和外部连接参数
- 上传文件、附件或业务侧依赖的持久化文件
- 部署脚本、镜像标签和版本记录

如果团队只备份数据库，而没有备份配置和部署信息，恢复时通常仍然会卡住。

## 监控时优先关注什么

### 平台可用性

- 登录页和平台首页是否可访问
- 核心管理入口是否可进入
- 平台服务是否持续重启或退出

### 基础依赖

- 数据库连接是否稳定
- RabbitMQ 是否可用
- Docker API 或容器运行环境是否正常
- 磁盘、内存和网络是否达到风险阈值

### 应用运行状态

- 业务应用能否正常启动
- 发布动作是否成功
- 应用日志中是否出现持续性错误

## 建议保留的日志与记录

- 平台服务日志
- 业务应用运行日志
- 业务日志与关键操作审计记录
- 关键依赖组件日志
- 升级、发布、回滚和故障处理记录

这些信息越早规范，后续故障定位越快。

## 建议的排障顺序

遇到问题时，建议按层次排查，而不是一上来就重启所有组件：

1. 先确认访问入口是否异常
2. 再确认平台服务本身是否正常
3. 再检查数据库、消息和运行时依赖
4. 最后再看具体业务应用和最近变更

## 常见故障判断思路

### 平台打不开

优先检查：

- 访问地址、端口和网络入口
- 平台容器或进程是否存活
- 最近是否做过升级或配置调整

### 平台能打开，但关键入口报错

优先检查：

- 数据库连接
- RabbitMQ 连接
- 初始化配置是否被改动
- 日志里是否有持续报错

### 应用无法发布或运行

优先检查：

- Docker API 是否可用
- 运行容器是否成功创建
- 应用相关配置是否齐全
- 平台和业务应用日志是否出现依赖异常

## 团队层面的建议做法

- 在正式环境中，不要只依赖“重启试试”
- 为平台和业务应用分别保留最基本的日志与状态检查入口
- 为关键业务操作建立业务日志和审计回溯路径
- 记录常见问题和团队内部处理流程
- 定期验证备份能否恢复，而不只是“有备份文件”
- 为升级、回滚和故障响应指定明确负责人
