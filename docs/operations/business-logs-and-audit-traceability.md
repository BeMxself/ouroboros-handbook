# 业务日志与审计回溯

这页适合已经进入正式交付或正式运行阶段的团队。

很多团队在排障时只看服务日志，但真正需要回答的问题往往是：

- 谁在什么时候做了什么操作
- 这次操作影响了哪个业务对象
- 结果是成功、失败，还是部分异常
- 当时传入了什么关键数据

这些问题并不属于通用技术日志的强项，而是更适合交给业务日志能力来承接。

## 先分清楚三类日志

### 平台服务日志

适合排查应用启动、依赖连接、异常堆栈和运行时错误。

### 依赖组件日志

适合排查数据库、RabbitMQ、网关、容器等基础设施问题。

### 业务日志

适合回答“某个业务操作到底发生了什么”。

潮汐栈当前的 `ouroboros-business-log` 记录的是 HTTP 侧业务动作，不是通用链路追踪，也不是简单访问日志。

## 业务日志当前能记录什么

从当前实现看，业务日志至少可以沉淀这些信息：

- 模块名
- 操作名
- 操作级别
- 操作人和关联员工
- 操作时间
- 业务键
- 摘要文本
- 操作结果
- 请求载荷
- 响应内容
- IP 地址和浏览器信息

这意味着它很适合用于审计、问题回溯和业务操作复盘。

## 它是怎么产生的

当前运行链路大致是这样：

1. 业务日志过滤器拦截 HTTP 请求
2. 系统按 HTTP 方法和 URL 匹配日志模板
3. 日志提取器从路径参数、查询参数、请求体和响应体里抽取数据
4. 按模板规则生成摘要、业务键和脱敏后的日志内容
5. 结果写入业务日志存储，并通过平台入口提供查询

这条链路里，最关键的不是“记不记日志”，而是“哪些操作该记、记成什么样、哪些字段必须脱敏”。

## 日志模板解决什么问题

业务日志模板控制的是记录规则，而不是日志展示样式。

当前模板能力已经支持：

- 按 HTTP 方法和 URL 匹配操作
- 定义模块名和操作名
- 指定业务键字段
- 配置摘要模板
- 配置敏感字段脱敏
- 决定是否保存响应体

这意味着团队不需要把所有日志规则硬编码在过滤器里，而可以按业务重要性逐步治理。

## 什么时候最值得启用业务日志

- 有审批、配置变更、权限调整、关键单据处理这类可追责操作
- 经常需要回查“某条业务记录是谁改的”
- 需要把操作结果和业务对象关联起来
- 需要向实施、运营或审计同学提供可读的业务操作记录

如果你们已经进入多人协作、跨角色使用或正式环境运行阶段，业务日志通常很有价值。

## 什么时候不能只靠业务日志

- 服务启动失败、线程异常、连接池耗尽这类技术问题
- 数据库、消息队列、网关或容器本身的故障
- 不经过 HTTP 入口的底层运行异常

这时仍然要结合平台服务日志、依赖日志和监控一起看。

## 模板设计时的几个实用原则

### 业务键要稳定

如果一个操作很关键，尽量为它选一个稳定、能定位业务对象的键，例如单据号、流程实例号、配置项名称，而不是临时字段。

### 摘要要让非开发也能看懂

摘要不是给机器看的，更适合写成“谁对什么对象执行了什么动作”的风格，方便实施和运营同学直接使用。

### 敏感字段要尽早脱敏

模板已经支持敏感字段脱敏，所以像密码、密钥、令牌、身份证号、手机号等内容不应直接裸写进日志。

### 不要默认保存所有响应体

只有在确实需要审计回放或回溯结果时，才建议保留响应体；否则会带来存储和敏感信息治理压力。

## 运维和实施阶段怎么用这组能力

### 做问题回溯

当用户反馈“操作过但结果不对”时，先看业务日志，确认：

- 是否真的发生了对应操作
- 操作时间和操作人是否匹配
- 业务键是否正确
- 返回结果是成功还是失败

### 做变更审计

对于权限、配置、审批和重要业务单据，业务日志可以帮助你快速回到“谁做了这次变更”。

### 做排障串联

业务日志适合做第一层定位，然后再继续串平台服务日志、依赖日志和最近变更记录。

## 团队落地建议

- 先从关键业务操作开始，不要一开始就追求全覆盖
- 为关键操作统一业务键命名方式
- 明确哪些字段必须脱敏
- 定期检查业务日志是否还在持续写入
- 把业务日志查看和故障处理流程放进团队值班或运维手册
